# OPA Policy Generation for MCP Agent

## Overview

This skill generates an OPA policy for an MCP-based agent. The target <TARGET_AGENT_PATH> is specified in the `.env` file.

**Inputs (all located in `<TARGET_AGENT_PATH>/smith/`):**
1. `guidance.txt` — natural language policy rules
2. `tool_definitions.json` — MCP tool list with parameters (generated by `smith --flag get_mcp_parameter`)
3. `system_vars.json` — system/session variables (user role, team, etc.) that become `input.extensions.subject.*`
4. `test_case_template.json` (this one locates in `./references/` not `<TARGET_AGENT_PATH>/smith/`) — defines the OPA input envelope structure

**Outputs:**
1. **OPA policy** (`package mcp.policies`) — saved to `<TARGET_AGENT_PATH>/smith/policy_generated.rego`
2. **Extension variables list** — saved to `<TARGET_AGENT_PATH>/smith/extension_variables.json`

---

## Prerequisites

Run `smith --flag get_mcp_parameter` first to generate `tool_definitions.json`. This connects to the MCP server and extracts all tool names, parameters, types, and descriptions.

---

## Step 1: Read the Input Data Structure

Read `test_case_template.json` to understand the OPA input envelope format:

```json
{
    "kind": "tool_call",
    "action": "execute",
    "name": "<tool_name>",
    "extensions": {
        "subject": {
            "claims": {}
        },
        "headers": {...},
        "labels": ["internal"],
        "agent": {
            "input": "<original user prompt>",
            "session_id": "...",
            "conversation_id": "...",
            "turn": <int>
        },
        "object": {
            "managed_by": "tool",
            "trust_domain": "internal"
        }
    },
    "arguments": {
        // MCP tool parameters go here
    }
}
```

The policy has access to three sources of data:

1. **`input.name`** — which tool is being called
2. **`input.arguments.<param>`** — MCP tool parameters (from `tool_definitions.json`)
3. **`input.extensions.subject.*`** — extension variables, which include:
   - **System variables** (from `system_vars.json`) — always pre-populated (e.g., roles, teams, id, approval)
   - **Additional extensions** — any new variables needed by guidance that are not already in `system_vars.json` or tool arguments

System variables ARE extensions — they live under `input.extensions.subject.*`. The only difference is that system variables are always available, while additional extensions must be explicitly declared in `extension_variables.json`.

---

## Step 2: Read Guidance, System Variables, and Identify Extension Variables

Read `guidance.txt`, `tool_definitions.json`, and `system_vars.json`. For each guidance rule, determine:

1. **Which MCP tool does it apply to?** (map to tool name from `tool_definitions.json`)
2. **Can this rule be enforced using only the tool's arguments?** (e.g., checking `amount > 200` uses `input.arguments.amount`)
3. **Does this rule require context NOT in the tool arguments/system variables?** If yes, identify the extension variable needed.

**Important:** Variables already defined in `system_vars.json` (e.g., `roles`, `teams`, `id`, `approval`) are always available in the OPA input as `input.extensions.subject.*`. Do NOT list them in `extension_variables.json` — they are pre-populated by the system and do not need to be declared as additional extensions.

Only add to `extension_variables.json` variables that:
- Are required by the guidance rules AND
- Are NOT already defined in `system_vars.json` AND
- Cannot be derived from tool arguments

If all required context variables are already covered by `system_vars.json` and tool arguments, `extension_variables.json` should contain an empty list:

```json
{
    "extension_variables": [],
    "note": "All required context variables are already defined in system_vars.json"
}
```

If there ARE variables needed beyond what `system_vars.json` provides:

```json
{
    "extension_variables": [
        {
            "path": "input.extensions.subject.<new_variable>",
            "type": "string",
            "candidates": ["value1", "value2"],
            "reason": "Guidance requires this context not available in system_vars.json or tool arguments",
            "rules_requiring": [1, 2]
        }
    ]
}
```

**Ask user for confirmation before proceeding.**

---

## Step 3: Generate the OPA Policy

Using `guidance.txt`, `tool_definitions.json`, and results from Step 2, generate the OPA policy. The policy has access to:

1. **`input.arguments.*`** — MCP tool parameters (from `tool_definitions.json`)
2. **`input.extensions.subject.*`** — includes:
   - System variables (from `system_vars.json`, always available — e.g., roles, teams, id, approval)
   - Additional extension variables (from Step 2, only if guidance requires context not in system variables or tool arguments)

Follow this structure:

```rego
package mcp.policies

default allow := false

# === Input Accessors ===
subject := input.extensions.subject
args := object.get(input, "arguments", {})

# === Constants ===
# (blocked values, thresholds — extract from guidance)

# === Tool Argument Keys ===
allowed_arg_keys := {
    "<tool_name>": {"<param1>", "<param2>", ...},
}

# === Global DENY/ALLOW Rules ===
# (rules that apply across all tools)

# === Tool-Specific DENY/ALLOW Rules ===
# (one deny block per guidance rule)

# === Final ALLOW ===
allow if {
    valid_envelope
    not any_deny
}
```

### Key Principles:
- **Prefer `input.arguments.*`** for tool-specific checks
- **Only use `input.extensions.*`** for context the tool doesn't provide
- **Each guidance rule → one `deny[msg]` block** with a clear error message
- **Reference values** (blocked domains, forbidden fields) become Rego set constants

**Save to:** `<TARGET_AGENT_PATH>/smith/policy_generated.rego`

---

## Step 4: Validate the Policy

Run:
```bash
python scripts/policy_generation/validate_policy.py \
    --policy <TARGET_AGENT_PATH>/smith/policy_generated.rego
```

If validation fails, fix errors and re-run. Use `--fix` to auto-format:
```bash
python scripts/policy_generation/validate_policy.py \
    --policy <TARGET_AGENT_PATH>/smith/policy_generated.rego --fix
```

---

## Step 5: Present Final Outputs

1. **OPA Policy** — `<TARGET_AGENT_PATH>/policy_generated.rego`
2. **Extension Variables** — `<TARGET_AGENT_PATH>/extension_variables.json` (do not include system variables in this file)

---

## Rules

- Do NOT add extension variables unless guidance explicitly requires context beyond tool arguments.
- Do NOT invent rules not in the guidance. Each deny rule must trace back to a specific guidance line.
- Do NOT modify any existing files. All outputs go to new files.
- Always validate the policy compiles before presenting it to the user.
- If guidance is ambiguous, ask the user for clarification before generating the rule.
