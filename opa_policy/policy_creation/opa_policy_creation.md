# OPA Policy Generation for MCP Agent

## Overview

This skill generates an OPA policy for an MCP-based agent. The target <TARGET_AGENT_PATH> is specified in the `.env` file.

**Inputs (all located in `<TARGET_AGENT_PATH>/smith/`):**
1. `guidance.txt` — natural language policy rules
2. `tool_definitions.json` — MCP tool list with parameters (generated by `smith --flag get_mcp_parameter`), the parameters become `input.arguments.*`
3. `system_vars.json` — system/session variables (user role, team, etc.) that become `input.extensions.subject.*`
4. `test_case_template.json` (this one locates in `./references/` not `<TARGET_AGENT_PATH>/smith/`) — defines the OPA input envelope structure

**Outputs:**
1. **OPA policy** (`package mcp.policies`) — saved to `<TARGET_AGENT_PATH>/smith/policy_generated.rego`
2. *(Optional)* **Extension suggestions** — saved to `<TARGET_AGENT_PATH>/smith/extension_suggestions.json` (informational only, not part of the policy)

---

## Prerequisites

Run `smith --flag get_mcp_parameter` first to generate `tool_definitions.json`. This connects to the MCP server and extracts all tool names, parameters, types, and descriptions.

---

## Step 1: Read the Input Data Structure

Read `test_case_template.json` to understand the OPA input envelope format:

```json
{
    "kind": "tool_call",
    "action": "execute",
    "name": "<tool_name>",
    "extensions": {
        "subject": {
            "claims": {}
        },
        "headers": {...},
        "labels": ["internal"],
        "agent": {
            "input": "<original user prompt>",
            "session_id": "...",
            "conversation_id": "...",
            "turn": <int>
        },
        "object": {
            "managed_by": "tool",
            "trust_domain": "internal"
        }
    },
    "arguments": {
        // MCP tool parameters go here
    }
}
```

The policy is designed based on two primary data sources:

1. **`input.name`** — which tool is being called
2. **`input.arguments.<param>`** — MCP tool parameters (from `tool_definitions.json`)
3. **`input.extensions.subject.*`** — system variables (from `system_vars.json`) — always pre-populated (e.g., roles, teams, id, approval)

---

## Step 2: Read Guidance, System Variables, and Tool Definitions

Read `guidance.txt`, `tool_definitions.json`, and `system_vars.json`. For each guidance rule, determine:

1. **Which MCP tool does it apply to?** (map to tool name from `tool_definitions.json`)
2. **Can this rule be enforced using the tool's arguments and system variables?** (e.g., checking `amount > 200` uses `input.arguments.amount`; checking role uses `input.extensions.subject.roles`)

The policy MUST only reference data available from:
- `input.arguments.*` — tool parameters defined in `tool_definitions.json`
- `input.extensions.subject.*` — system variables defined in `system_vars.json`

If a guidance rule requires context that is NOT available in either tool arguments or system variables, do NOT add it to the policy. Instead, log it as a suggestion in `extension_suggestions.json` (see Step 4).

**Ask user for confirmation before proceeding.**

---

## Step 3: Generate the OPA Policy

Using `guidance.txt`, `tool_definitions.json`, and `system_vars.json`, generate the OPA policy. The policy ONLY uses:

1. **`input.arguments.*`** — MCP tool parameters (from `tool_definitions.json`)
2. **`input.extensions.subject.*`** — system variables (from `system_vars.json`, always available — e.g., roles, teams, id, approval)

Follow this structure:

```rego
package mcp.policies

default allow := false

# === Input Accessors ===
subject := input.extensions.subject
args := object.get(input, "arguments", {})

# === Constants ===
# (blocked values, thresholds — extract from guidance)

# === Tool Argument Keys ===
allowed_arg_keys := {
    "<tool_name>": {"<param1>", "<param2>", ...},
}

# === Global DENY/ALLOW Rules ===
# (rules that apply across all tools)

# === Tool-Specific DENY/ALLOW Rules ===
# (one deny block per guidance rule)

# === Final ALLOW ===
allow if {
    valid_envelope
    not any_deny
}
```

### Key Principles:
- **Use `input.arguments.*`** for tool-specific checks
- **Use `input.extensions.subject.*`** for system variable checks (roles, teams, etc.)
- **Each guidance rule → one `deny[msg]` block** with a clear error message
- **Reference values** (blocked domains, forbidden fields) become Rego set constants

**Save to:** `<TARGET_AGENT_PATH>/smith/policy_generated.rego`

---

## Step 4: Log Extension Suggestions (Optional)

If any guidance rules could NOT be enforced because they require context not available in system variables or tool arguments, save suggestions to `<TARGET_AGENT_PATH>/smith/extension_suggestions.json`:

```json
{
    "suggestions": [
        {
            "guidance_rule": "Rule text from guidance.txt",
            "missing_context": "Description of what data is needed",
            "suggested_path": "input.extensions.subject.<suggested_variable>",
            "type": "string",
            "candidates": ["value1", "value2"],
            "reason": "This rule cannot be enforced without additional context not in system_vars.json or tool arguments"
        }
    ],
    "note": "These are suggestions for future system variable additions. They are NOT part of the current policy."
}
```

If all guidance rules can be fully enforced with existing system variables and tool arguments, skip this file.

---

## Step 5: Validate the Policy

Run:
```bash
python scripts/policy_generation/validate_policy.py \
    --policy <TARGET_AGENT_PATH>/smith/policy_generated.rego
```

If validation fails, fix errors and re-run. Use `--fix` to auto-format:
```bash
python scripts/policy_generation/validate_policy.py \
    --policy <TARGET_AGENT_PATH>/smith/policy_generated.rego --fix
```

---

## Step 6: Present Final Outputs

1. **OPA Policy** — `<TARGET_AGENT_PATH>/smith/policy_generated.rego`
2. *(If applicable)* **Extension Suggestions** — `<TARGET_AGENT_PATH>/smith/extension_suggestions.json` (informational only — these are NOT enforced by the policy)

## Step 7: Human Approval

Ask users if they are satisfied with the policy. If not, ask them to provide feedback. If yes, `cp <TARGET_AGENT_PATH>/smith/policy_generated.rego ./assets/opa/policy.rego`.

---

## Rules

- The policy MUST only use data from `system_vars.json` and `tool_definitions.json`. Do NOT design the policy around hypothetical extension variables.
- Do NOT invent rules not in the guidance. Each deny rule must trace back to a specific guidance line.
- Do NOT modify any existing files. All outputs go to new files.
- Always validate the policy compiles before presenting it to the user.
- If guidance is ambiguous, ask the user for clarification before generating the rule.
- If a guidance rule cannot be enforced with available data, log it as a suggestion — do NOT add phantom variables to the policy.
